# 荐股算法流程说明

本文档基于当前仓库实现，概括系统如何从行情数据出发，生成每日推荐股票池与组合权重，并说明 LLM 新闻链路在整体体系中的位置。

## 1. 总体目标

系统的核心任务不是预测某一只股票的精确涨幅，而是在每个交易日收盘后，从全市场中筛出一组相对更优、风险更可控的候选标的。

主流程可概括为：

```text
行情数据 -> 股票池预过滤 -> 因子计算 -> 截面打分/模型排序 -> Top-K 推荐池
        -> 次日可买性过滤 -> 组合权重生成 -> 推荐结果输出 -> 事后评估
```

## 2. 输入数据

主线荐股算法使用日线量价数据，主要字段包括：

- 开盘价 `open`
- 最高价 `high`
- 最低价 `low`
- 收盘价 `close`
- 成交量 `volume`
- 换手率 `turnover`

数据默认通过 `AkShare` 拉取，并增量写入本地 `DuckDB`。统一的结果输出目录由 `config.yaml` 中的 `paths.results_dir` 控制，默认是 `data/results`。

## 3. 执行时点与交易口径

系统默认在收盘后运行，用当日及以前的数据生成信号，因此不会使用未来信息。

默认交易口径为：

1. `t` 日收盘后生成推荐。
2. 最早在 `t+1` 日开盘买入。
3. 按配置的持有期做后续评估。

这也是为什么系统更强调“收盘后选股”和“次日可交易性”，而不是盘中追涨。

## 4. 主流程分解

### 4.1 股票池准备

系统先确定待处理股票列表：

- 若命令行传入 `--symbols`，则只处理指定股票。
- 否则从默认股票池中读取全市场或缓存股票列表。
- 若未使用 `--skip-fetch`，会先执行一次日线增量更新。

这一阶段的目标是保证后续因子计算使用的是最新、可追溯的数据。

### 4.2 数据质量检查

写入完成后，系统会检查：

- 行情缺失情况
- OHLC 合法性
- 交易日跨度是否异常

若数据质量存在问题会记录告警，但不会轻易中断整个流程，避免因为单一数据源抖动影响全量跑批。

### 4.3 股票池预过滤

在真正打分前，系统会先做一层硬规则过滤，避免把明显不可取的标的交给模型排序。当前配置重点控制以下风险：

- 近几日涨停或跌停次数过多
- 换手率过低或过高
- 价格处于长周期区间顶部
- 其他高博弈、高波动特征

这一步的核心作用是先排除“妖股化”“死水股化”“绝对高位化”的候选，降低模型误选概率。

## 5. 因子计算

系统以宽表形式构造历史价格矩阵，再使用 GPU 或 CPU 计算因子。当前主线因子可分为四类。

### 5.1 趋势与强弱因子

- 动量 `momentum`
- RSI `rsi`

它们用于识别价格延续性与短中期强弱状态。

### 5.2 波动与流动性因子

- ATR `atr`
- 实现波动率 `realized_vol`
- 换手率均值 `turnover_roll_mean`
- 波动率/换手率比值 `vol_to_turnover`
- 成交量偏度 `volume_skew_log`

这类因子主要描述“波动是否过大”“交易是否过热”以及筹码结构是否稳定。

### 5.3 量价关系因子

- 量价相关 `vol_ret_corr`
- 短期反转 `short_reversal`
- 量价趋势 `volume_price_trend`

这类因子用于区分“健康放量上涨”和“情绪推动式脉冲行情”。

### 5.4 高位约束与 K 线结构因子

- 短中期乖离率 `bias_short`、`bias_long`
- 近端涨幅 `recent_return`
- 最大单日跌幅 `max_single_day_drop`
- 区间位置 `price_position`
- 市值代理 `log_market_cap`
- 日内振幅、上下影线、隔夜跳空、尾盘强度等 K 线结构因子

这一组因子是当前系统抑制高位追涨、降低妖股暴露的重要手段。

## 6. 打分与排序

系统支持多种排序方式，由 `config.yaml` 中的 `signals.sort_by` 控制。

### 6.1 线性组合模式

- `momentum`
- `rsi`
- `composite`
- `composite_extended`

其中：

- `composite` 主要按动量和 RSI 的加权组合打分。
- `composite_extended` 会引入更多波动、量价、高位约束和 K 线结构因子。

### 6.2 机器学习模式

- `xgboost`：当前默认主线
- `deep_sequence`：基于 OHLCV 序列的深度学习模型

默认情况下，系统会优先走 `xgboost`，使用预先训练好的树模型工件，对最新截面因子进行打分排序。

## 7. 大盘状态切换

当排序方式是 `composite` 或 `composite_extended` 时，系统可启用 `Regime Switch`：

- 牛市：增强趋势类权重
- 熊市：增强低波动、大市值、防守类权重
- 震荡市：提高反转类因子的相对重要性

这一步相当于在最外层加了一个市场环境适配器，让同一套因子在不同市场状态下表现得更稳健。

## 8. 生成推荐池

排序完成后，系统会：

1. 按分数从高到低排序。
2. 先取一个放大的候选池。
3. 做“次日可买性过滤”。
4. 最终截取 `Top-K` 作为推荐结果。

其中“次日可买性过滤”主要用于剔除：

- 停牌标的
- 次日开盘一字涨停等难以成交的标的

因此，最终输出的不只是“分数高”，还要求“次日更可能买得到”。

## 9. 组合权重生成

推荐池生成后，系统不会默认等权，而是可按配置生成更稳健的组合权重。当前支持：

- `equal`：等权
- `score`：按分数归一化
- `risk_parity`：风险平价
- `min_variance`：最小方差
- `mean_variance`：均值-方差

默认配置偏向使用风险平价或最小方差思路，并叠加如下约束：

- 单票权重上限
- 行业权重上限
- 最大换手约束
- 协方差收缩与 ridge 稳定化

这一步的目标是把“选股结果”转化为“可执行组合”，并控制单一高波动股票对净值的伤害。

## 10. 输出结果

主线荐股流程最终会输出 `recommend_*.csv`，通常包含：

- 股票代码
- 各类因子值
- 排序分数
- 推荐理由
- 权重
- 排序方式
- 市场状态标签 `regime`
- 截止交易日 `asof_trade_date`

这些文件默认写入 `data/results/`。

## 11. 事后评估

系统还提供评估链路，对历史推荐结果做前向收益标注与组合表现分析，常见用途包括：

- 查看单票未来收益
- 评估组合收益与交易成本影响
- 比较不同排序方式或参数配置

这部分用于回答“策略历史上是否有效”，而不是直接参与当天选股。

## 12. LLM 新闻链路的定位

仓库中还有一条独立的 LLM 链路：

```text
东方财富飙升榜/人气榜 -> 拉取个股新闻 -> LLM 判断是否为真实新闻驱动
                   -> 输出新闻驱动股票清单 -> 生成 Markdown/HTML 报告
```

这条链路的作用是：

- 从“关注度飙升”股票中筛出真正由新闻催化驱动的标的
- 区分新闻事件与纯市场波动
- 为人工复盘或事件驱动研究提供补充信息

需要注意的是：它目前是独立输出的辅助分析模块，尚未直接并入主线量化荐股打分体系。

## 13. 一句话总结

如果把整个系统用一句话概括，就是：

> 先用日线量价数据和风险约束筛掉不合适的股票，再用因子与模型做截面排序，最后生成可交易、可控风险的推荐池和组合权重。

## 14. 常用命令

### 主线荐股

```bash
python scripts/daily_run.py --max-symbols 100
python scripts/daily_run.py run --skip-fetch --top-k 20
python scripts/daily_run.py --symbols 600519,000001
```

### 推荐结果评估

```bash
python scripts/daily_run.py eval --latest --horizon 5
python scripts/daily_run.py eval --csv data/results/recommend_2026-03-27.csv
```

### LLM 新闻辅助分析

```bash
python scripts/llm_daily_analysis.py --with-macro
python scripts/llm_analysis_report.py --latest
```

## 15. 风险提示

本系统输出的是研究与筛选结果，不构成任何投资建议。实际交易仍需结合流动性、行业事件、公告时点、仓位管理与个人风险承受能力独立判断。
