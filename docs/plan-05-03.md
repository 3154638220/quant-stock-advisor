## 一、严重缺陷（直接影响回测与推理正确性）

### 1. `src/market/regime.py` — 累积收益用简单求和而非复利

**问题代码（`classify_regime`）：**
```python
short_return = float(np.sum(short_rets))   # ❌ 简单加总
long_return  = float(np.sum(long_rets))
```
简单加总在窗口较长（20~60个交易日）时，与真实复利累积收益误差高达 1～3%，导致 `bull_return_threshold`（0.05）和 `bear_return_threshold`（0.04）的阈值对齐失准，regime 分类出错。

**修复方案：**
```python
short_return = float(np.prod(1.0 + short_rets) - 1.0)
long_return  = float(np.prod(1.0 + long_rets)  - 1.0)
```

### 2. `src/models/baseline/train.py` — 随机 `train_test_split` 引入前视偏差

**问题代码：**
```python
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=test_size, random_state=seed)
```
量化数据具有时序依赖性，随机划分会导致用未来数据训练/评估过去——即"未来函数"。这是量化回测中最典型的过拟合来源。

**修复方案：** 改用 `TimeSeriesSplit` 或固定时间截点：
```python
from sklearn.model_selection import TimeSeriesSplit
# 或按时间列排序后按比例切割：
cutoff = int(len(X) * (1 - test_size))
X_train, X_test = X[:cutoff], X[cutoff:]
y_train, y_test = y[:cutoff], y[cutoff:]
```
同时建议将 `test_size` 参数重命名为 `val_fraction`，避免混淆。

### 3. `src/models/inference.py` — rank\_ic 翻转逻辑使用训练期数据，危险

**问题代码：**
```python
rank_ic = metrics.get("val_rank_ic") or metrics.get("train_rank_ic")
if rank_ic is not None and rank_ic < 0:
    pred = -pred  # 自动翻转
```
训练期 `val_rank_ic < 0` 表示模型在训练集上方向错误，但在生产推理中直接翻转分数，没有考虑：① regime 切换后方向可能已经正确；② 翻转本身未经线上验证。这是一个隐性的逻辑错误，容易在牛熊切换时做反。

**修复方案：** 移除自动翻转，改为在训练后的模型评审阶段（`ic_f1_gate`）强制拒绝 rank_ic 持续为负的模型上线，而不是推理时静默修正：
```python
# 训练阶段：如果验证集 rank_ic < 0，直接拒绝保存该 bundle
if val_rank_ic < 0:
    raise ValueError(f"模型 val_rank_ic={val_rank_ic:.4f} < 0，拒绝保存，请检查特征或标签方向")
```

### 4. `src/backtest/transaction_costs.py` — 换手成本双边计算逻辑有偏差

**问题代码：**
```python
def turnover_cost_drag(turnover_half_l1: float, costs: TransactionCostParams) -> float:
    one_way = (costs.commission_buy_bps + costs.slippage_bps_per_side) * 1e-4
    sell_extra = (costs.stamp_duty_sell_bps + costs.commission_sell_bps - costs.commission_buy_bps) * 1e-4
    per_unit = 2.0 * one_way + max(0.0, sell_extra)  # ❌
    return t * per_unit
```
`turnover_half_l1 = 0.5 * Σ|w_new - w_old|` 表示单边换手量（买方或卖方各占 `t`），成本应为 `t * (buy_cost + sell_cost)`，而不是 `t * (2 * one_way + sell_extra)`。当前公式对佣金双倍计算，对印花税则低估。

**修复方案：**
```python
per_unit = costs.buy_fraction() + costs.sell_fraction()
return t * per_unit  # t 本身是单边，两侧均施加一次
```

## 二、重要问题（一致性与鲁棒性）

### 5. `src/features/standardize.py` 与 `src/features/rank_score.py` 的 zscore 实现不一致

`standardize.py` 的 `zscore_cross_section` 使用 `s.mean()` 和 `s.std(ddof=0)`（pandas 默认 `skipna=True`，但未显式排除 NaN 后再算），而 `rank_score.py` 的 `cross_section_zscore` 用 `np.nanmean` / `np.nanstd`。两处实现**共存于同一管道**，在含 NaN 时结果不等，导致因子标准化在不同阶段产生差异。

**修复方案：** 在 `src/features/standardize.py` 中建立统一实现，并在 `rank_score.py` 中导入复用：
```python
# standardize.py (统一定义)
def zscore_cross_section(s: pd.Series, *, eps: float = 1e-12) -> pd.Series:
    arr = pd.to_numeric(s, errors="coerce").to_numpy(dtype=np.float64)
    valid = np.isfinite(arr)
    if not valid.any():
        return pd.Series(np.nan, index=s.index, dtype=float)
    m = np.nanmean(arr)
    sd = np.nanstd(arr, ddof=0)
    ...
```
`rank_score.py` 中 `cross_section_zscore` 改为从 `src.features.standardize` 导入，消除重复。

### 6. `src/features/factor_eval.py` — IC 信息比率（IR）用总体标准差，被轻微高估

**问题代码（`ic_summary`）：**
```python
s = float(x.std(ddof=0))   # ❌ 总体标准差
ir = m / s
```
而同文件外 `performance_panel.py` 的 `sharpe_ratio` 用 `ddof=1`（样本标准差）。两处不一致，且量化领域 IR 均约定使用样本标准差。当 IC 序列较短时（n=12~24个月），`ddof=0` 会低估标准差约 2～5%，导致 IR 虚高。

**修复方案：**
```python
s = float(x.std(ddof=1))  # ✅ 样本标准差，与夏普口径一致
```

### 7. `src/features/neutralize.py` — 小截面回归退化阈值过于宽松

**问题代码：**
```python
if m.sum() < X.shape[1] + 1:
    residuals.loc[idx] = y - (_safe_nanmean(y[m]) if m.any() else 0.0)
    continue
```
当截面中有效行数略大于特征维数时（如 `X.shape[1]=25`，`valid=27`），仍强行做 lstsq 回归，矩阵条件数极差，残差无统计意义。应使用更保守的条件。

**修复方案：**
```python
min_samples = max(20, X.shape[1] * 3)   # 至少3倍于特征数
if m.sum() < min_samples:
    residuals.loc[idx] = y - _safe_nanmean(y[m])
    continue
# 额外检查条件数
XtX_cond = np.linalg.cond(X[m])
if XtX_cond > 1e8:
    residuals.loc[idx] = y - _safe_nanmean(y[m])
    continue
```

### 8. `src/market/regime.py` — 因子名用子串匹配 `if key in factor_name`，易错配

**问题代码（`get_regime_weights`）：**
```python
BULL_MULTIPLIERS = {
    "momentum": 1.5,
    "reversal": 0.3,
    ...
}
def _pick_multiplier(factor_name: str, mapping):
    for key, m in mapping.items():
        if key in factor_name:   # ❌ 子串匹配
            return float(m)
```
若配置中出现 `momentum_12_1`、`reversal_ratio`、`bias_long` 等因子，`"long"` 会匹配到 `bias_long` 的 `log_market_cap`（BEAR 中 `"bias_long"` 和 `"log_market_cap"` 均含 `"a"`），顺序敏感且不可预测。

**修复方案：** 改为精确全词匹配或使用显式配置映射：
```python
# 精确前缀匹配
def _pick_multiplier(factor_name: str, mapping):
    # 先精确匹配，再前缀
    if factor_name in mapping:
        return float(mapping[factor_name])
    for key, m in mapping.items():
        if factor_name.startswith(key + "_") or factor_name == key:
            return float(m)
    return 1.0
```
同时在 `config.yaml` 的 `regime` 节下新增 `factor_group_map` 字段，将因子名和组别的对应关系显式声明。

## 三、设计改进（扩展性与长期维护）

### 9. `src/features/ic_monitor.py` — 全量 JSON 加载存在性能瓶颈

**问题：** `_load_store` / `_save_store` 每次 `append` 都将整个 JSON 全量读写（load → modify list → dump）。当积累数万条记录（约 250 因子 × 500 交易日 = 125,000 行）时，单次操作耗时将超过 10 秒。

**修复方案：** 改为 JSONL（每行一条）追加写入，读取时按需 `grep` 或全量流式解析，与 `experiment.py` 中 `append_experiment_jsonl` 对齐：
```python
def append(self, ...):
    record = ICRecord(...)
    with self._store_path.open("a", encoding="utf-8") as f:
        f.write(json.dumps(asdict(record)) + "\n")

def load_all(self) -> list[ICRecord]:
    records = []
    if self._store_path.exists():
        for line in self._store_path.read_text().splitlines():
            records.append(ICRecord(**json.loads(line)))
    return records
```

### 10. `src/models/research_contract.py` — `result_id` 缺少时间戳，存在碰撞风险

**问题代码：**
```python
def build_result_id(identity, data_slices, metrics) -> str:
    primary_slice = data_slices[0].slice_hash()[:12]
    metrics_hash = stable_hash(metrics, prefix_len=12)
    return f"{identity.research_topic}:{identity.research_config_id}:{primary_slice}:{metrics_hash}"
```
若两次不同时间运行的实验恰好产生相同 metrics（如均为 `{}`），则 `result_id` 完全相同，研究追踪系统将静默覆盖历史记录。

**修复方案：** 引入时间戳或 UUID 短码：
```python
def build_result_id(identity, data_slices, metrics, *, created_at: str | None = None) -> str:
    primary_slice = data_slices[0].slice_hash()[:10] if data_slices else "noslice"
    metrics_hash = stable_hash(metrics, prefix_len=10)
    ts_fragment = (created_at or utc_now_iso())[:16].replace(":", "").replace("-", "")  # 12位时间
    return f"{identity.research_topic}:{identity.research_config_id}:{primary_slice}:{metrics_hash}:{ts_fragment}"
```

### 11. `src/features/rank_score.py` — RSI 特殊处理硬编码因子名，扩展性差

**问题代码：**
```python
if name == "rsi":                      # ❌ 因子名硬编码
    z = cross_section_zscore(50.0 - r)
else:
    z = cross_section_zscore(raw)
```
同样的逻辑在 `cross_section_z_columns`、`composite_extended_linear_score`、`sort_key_for_dataframe` 中重复出现 3 次。一旦新增类 RSI 因子（如 `rsi_14`、`stoch_rsi`），需改多处。

**修复方案：** 引入 `mean_revert_factors: set[str]` 参数，由 `config.yaml` 注入：
```python
def cross_section_z_columns(
    df: pd.DataFrame,
    raw_names: Sequence[str],
    *,
    mean_revert_center: Mapping[str, float] | None = None,  # {"rsi": 50.0, "stoch_rsi": 50.0}
) -> pd.DataFrame:
    center_map = mean_revert_center or {}
    for name in raw_names:
        raw = pd.to_numeric(df[name], errors="coerce").to_numpy(dtype=np.float64)
        if name in center_map:
            z = cross_section_zscore(center_map[name] - raw)
        else:
            z = cross_section_zscore(raw)
        df[f"z_{name}"] = z
```

### 12. 整体架构 — 缺乏 walk-forward 验证框架

**问题：** 项目目前的验证链路是：因子 IC 评估 → 单一时间段回测 → 促进上线（`configs/promoted/`）。没有 walk-forward cross-validation 来验证模型在滚动窗口上的稳定性，导致：
- 无法区分模型是"真实 alpha"还是"参数过拟合于某段行情"
- `ic_f1_gate.py` 的门控仅基于全样本 IC 均值和 t 值，缺乏时序稳定性检验

**改进方案：** 在 `src/backtest/` 中新增 `walk_forward.py`，实现：
```python
@dataclass
class WalkForwardConfig:
    train_months: int = 24        # 训练窗口
    test_months: int = 3          # 验证窗口
    step_months: int = 1          # 滚动步长
    min_folds: int = 12           # 最小折数
    
def walk_forward_ic(
    df: pd.DataFrame,
    factor_col: str,
    forward_ret_col: str,
    cfg: WalkForwardConfig,
) -> pd.DataFrame:
    """返回每折的 IC 均值、IR、最大回撤 IC，用于稳定性评估。"""
    ...
```
然后将 `ic_f1_gate.py` 的准入条件扩展为：除了全样本 IC 外，还要求滚动 IC 的正比例（hit_rate）≥ 60%，以及滚动 IC 标准差 / |均值| ≤ 3（即 IR 的倒数低于 3）。

## 补充：一处小错误

`src/backtest/risk_metrics.py` 的 `max_drawdown_from_returns` 在输入序列含 `-1.0` 时（单日全仓清零），`equity = cumprod(1+r)` 会变为 0，后续 `1.0 - equity/peak` 全为 1.0，应在分母处添加 `np.maximum(peak, 1e-15)` 保护（代码已有，但 `equity` 为 0 时整条序列的回撤统计仍然有效）——这个其实没问题，已实现正确。真正需要加的是当 `equity` 出现负数（如杠杆策略，不适用于本系统）时的保护，可加一行注释说明此函数仅适用于非杠杆组合。
