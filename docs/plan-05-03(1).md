# 量化月度选股：不足诊断与后续计划

**文档角色**：基于 2026-05-03 代码快照的独立第三方诊断  
**诊断时间**：2026-05-03  
**核心结论**：当前**不宜启动新模型训练**；M10 成本压力测试的关键数字尚未产出，P1-3 评价体系存在不一致，P0-1 Step 2 行业内中性化未真正落地，这三项是启动下一轮训练的前置条件。

## 一、对"现在可以开始训练模型了吗"的直接回答

**结论：不能，原因有三个并列阻断项。**

| 阻断项 | 影响 |
|--------|------|
| **M10 cost gate 未通过**：30/50 bps 下 M8 natural 候选的 after-cost excess 数字不存在，promotion gate 的核心条件未验证 | 不知道信号在真实成本下是否仍有效 |
| **P1-3 Rank IC 不一致（`[~]` 进行中）**：同一模型两份报告 Rank IC 相差 4 倍（0.007 vs 0.027），评价体系不可信 | 当前无法判断哪个模型更好 |
| **P0-1 Step 2 行业内特征中性化未落地**：`monthly_ltr.py` 中无 `industry_neutral_zscore` 或 `_ind_z` 代码；plan-05-03 的修改方案是建议文字，未提交 | M6 XGBoost 仍对 `asset_turnover` 等行业异质因子有系统偏差，重训只会重复同样的行业集中问题 |

此外，如果"训练模型"是指时序深度学习模型（`src/models/timeseries/` 中的 LSTM/GRU/TCN/Transformer），则 `docs/plan.md` 第 11 节停止清单明确写明：**"不把深度学习作为当前第一主线"**，该方向在当前阶段冻结。

## 二、完整不足清单

### 2.1 M10 成本压力测试——核心阻断

**问题描述**：

`run_monthly_benchmark_suite.py` 中存在 `COST_SENSITIVITY_GRID`（10/30/50 bps）定义和 `build_cost_sensitivity()` 函数，但该函数的实现方式是**对 base=10bps 的 `cost_drag` 线性缩放**，而不是以不同 cost 参数重跑完整回测。具体表现：

```python
# 当前实现：cost_drag 按比例缩放（非真实重跑）
scaled_drag = base_drag * (bps / max(float(base_cost_bps), 1e-6))
```

这一方式的问题：月度换手率不是常数，买入失败概率和成交额也随月份变化，线性缩放会系统低估高成本场景的实际影响。

更严重的是：`monthly_selection_benchmark_suite_2026-05-01.md` 报告中**没有 cost sensitivity 表格**，说明 cost sensitivity CSV 虽然有输出路径定义，但其内容从未出现在可供审查的报告页面。

同时，benchmark suite 运行的模型是 `M8_regime_aware_fixed_policy__indcap3`（hard-cap），**M8 natural soft risk-budget 候选从未进入 benchmark suite**，因此 natural 候选在 30/50 bps 下的超额数字完全不存在。

**promotion gate 要求**：`after-cost excess 在 30bps 下 > 0`——当前无法判断。

**涉及文件**：
- `scripts/run_monthly_benchmark_suite.py` — `build_cost_sensitivity()`
- `src/pipeline/monthly_multisource.py` — `M5RunConfig.cost_bps`
- `scripts/run_monthly_selection_m8_natural_industry_constraints.py`

### 2.2 P1-3 Rank IC 数据不一致——评价体系失效

**问题描述**：

同一模型（M6 xgboost_rank_ndcg, U2_risk_sane, Top20）在两份报告中 Rank IC 数字矛盾：

| 报告 | 截止日 | Rank IC mean | Rank IC IR | 状态 |
|------|--------|-------------|-----------|------|
| M9 integrity (2026-04-30) | 2026-04-29 | 0.00666 | 0.086 | 极弱，几乎为 0 |
| M7 pitfix (2026-05-03) | 2026-04-30 | 0.02660 | 0.303 | 差 4 倍，不可解释 |

1 天数据差异（一个新月份标签）不可能带来 4 倍 IC 变化，说明两次运行使用了不同版本的特征集或不同的训练配置（`max_fit_rows=0` vs `max_fit_rows=50000` 的差异可能是原因）。

在这一不一致解决前，**任何"Rank IC 为正"的 gate 判断都不可靠**。

当前状态：`[~]` 进行中（plan-05-03 P1-3），未完成。

**涉及文件**：
- `scripts/run_monthly_selection_report.py`
- `scripts/run_monthly_selection_ltr.py`
- `src/models/research_contract.py` — config_snapshot / config_hash

### 2.3 P0-1 Step 2 行业内特征中性化未落地——代码与文档脱节

**问题描述**：

plan-05-03 的 P0-1 修改方案 Step 2 写了 `industry_neutral_zscore()` 函数的完整实现，声明状态为 `[x]`，但检查 `src/pipeline/monthly_ltr.py` 全文**不存在**以下任何字符串：

后果：M6 XGBoost ranker 对证券行业的 `asset_turnover` 特征仍保持系统性高权重，即使通过报告层 cap 限制了输出，模型本身的偏差没有修正。重新训练时该偏差会复现。

**涉及文件**：

### 2.4 M8 natural 候选未接入 benchmark suite——关键对比缺失

**问题描述**：

plan.md 第 0 节"当前决策"中的核心候选是：
```
U1 + M5 ElasticNet market_excess + Top20 + gamma0.20
after-cost 月均超额 ≈ 0.018338，Rank IC ≈ 0.103597
```

而 benchmark suite（`2026-05-01`）运行的是：
```
M8_regime_aware_fixed_policy__indcap3（hard-cap）
after-cost 月均超额 = 2.23%（benchmark suite 输出），Sharpe = 1.41
```

两者完全不同的模型，benchmark suite 中**没有 M8 natural soft 候选的对比行**。这导致：
- M8 natural 候选和 hard-cap baseline 之间没有在同一框架下比较
- cost sensitivity 即使跑出来也是针对 hard-cap 的，不代表 natural 候选的成本承受能力

### 2.5 时序模型代码悬空——技术债务

**问题描述**：

`src/models/timeseries/` 目录包含完整实现：
- `lstm_tcn.py`：LSTMRegressor, GRURegressor, TCNRegressor, TransformerEncoderRegressor

但在整个 `src/pipeline/`、`src/models/`、`scripts/` 目录中，**没有任何文件 import 或调用** `src.models.timeseries`。该模块是死代码，且与 stop list 第 10 条矛盾（"不把深度学习作为当前第一主线"）。

**风险**：该模块占用维护成本，且可能让新接触代码的人误以为时序模型是当前主线。

### 2.6 数据时效性滞后——信号已过期

**问题描述**：

当前最新可执行信号日为 `2026-03-31`，下一交易日为 `2026-04-01`。今天是 `2026-05-03`，意味着当前推荐名单已经**过期超过 30 天**，在实际使用中毫无意义。

原因：`2026-04-29` 日线数据缺少 `2026-04-30` 的下一交易日行情，因此不满足可执行信号日条件。

为使 `2026-04-30` 成为可执行信号日，需要：
1. 补齐 `2026-04-30` 全市场日线（`python scripts/fetch_only.py`）
2. 重跑 M2 dataset（更新 `candidate_pool_pass` + `next_trade_date`）
3. 重跑 M7 报告

**涉及文件**：
- `scripts/fetch_only.py`

### 2.8 Walk-forward 训练窗口过短 + fund_flow 历史覆盖短

**问题描述**：

- `min_train_months: 24`（最小训练窗口），即仅用 2 年历史训练模型
- fund_flow 数据起始日：`2023-10-01`，意味着在 2025-10 之前的所有训练窗口中，fund_flow 特征实际上是 0 覆盖（仅靠缺失标志 `is_missing_feature_fund_flow_*`）
- 总信号月 64 个，去掉最小训练窗口 24 个后，有效 OOS 样本约 40 个月

这意味着：带 fund_flow 特征的模型在 2018-2023 期间的"训练"实际上是在用全为 NaN→0 的特征训练，这会使模型对这些特征的权重学习到虚假零信号。

P1-2 虽然加了 `FAMILY_DATA_START` 按数据族调整训练起点，但实际上 M6 中**这一配置没有被激活**（`window_type="expanding"` 且 fund_flow 起始日没有影响 walk-forward 分割逻辑）。

## 三、后续计划（分阶段）

### Phase 0：解除阻断项（1-2 天）

在启动任何模型训练之前，必须完成以下三项。

**Task 0-A：落地 P0-1 Step 2 行业内特征中性化**

在 `src/pipeline/monthly_ltr.py` 中实装 `industry_neutral_zscore()`，对基本面特征族做行业内 z-score，替换 `_train_predict_xgboost_ranker` 中对应特征列的引用。

验收标准：
```python
# 新增单元测试（已在 plan-05-03 定义）
tests/test_monthly_selection_ltr.py::test_industry_neutral_zscore_reduces_sector_bias
```
重跑 M6 walk-forward，确认 2026-03-31 信号日 Top20 中单行业数量 ≤ 6（与 apply_industry_cap 双保险）。

**Task 0-B：解决 P1-3 Rank IC 不一致**

统一 M7 pitfix 与 M9 integrity 的配置，特别是 `max_fit_rows` 和 walk-forward 窗口，确认两次运行用同一 feature parquet。在 `ResearchManifest` 中记录 `feature_file_hash` 和 `config_hash`，保证可复现。

验收标准：同一模型、同一数据截止日，两次运行的 Rank IC 差异 < 10%。

**Task 0-C：产出 M8 natural 候选的真实 30/50 bps 结果**

用 `M5RunConfig(cost_bps=30.0)` 和 `cost_bps=50.0` 分别对 M8 natural 候选（`U1 + ElasticNet + Top20 + gamma0.20`）重跑完整回测，记录：
- `after_cost_excess_mean_30bps`
- `after_cost_excess_mean_50bps`
- `breakeven_bps`

不要用线性缩放替代。在 benchmark suite 中新增 M8 natural 候选行，与 hard-cap 并排比较。

验收标准：有明确的数字判断 `after_cost_excess_mean_30bps > 0`（pass）或 `≤ 0`（fail，则 M10 未通过，继续修复）。

### Phase 2：M10 cost gate 最终验收（2-3 天）

在 Phase 0 Task 0-C 完成后，根据结果执行：

**如果 30bps 通过**（`after_cost_excess > 0`）：

继续完成 M10 剩余测试项：
- 涨停买入失败压力：`limit_up_mode=idle` vs `redistribute` 对比
- 容量分析：推荐名单的 `amount_20d_mean` 分布，确认 20M 规模下不产生显著冲击
- 高换手月份单独披露（换手 > 80% 的月份标记）

输出：`monthly_selection_m10_cost_pressure_YYYY-MM-DD.md`，M10 gate 正式通过。

**如果 30bps 不通过**（`after_cost_excess ≤ 0`）：

说明当前模型信号强度不足以覆盖真实交易成本，需要回到特征改进：
- 优先接入北向资金和主题/概念 breadth（M11 优先队列前两项）
- 评估是否加大 `top_k=30` 而非 `top_k=20`（换手率下降）
- 评估更低频换仓（双月或季度）
