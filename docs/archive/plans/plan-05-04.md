## 1. 总体评价

### 优势

- **研究纪律严格**：PIT（Point-in-Time）口径贯穿全链路，信号日、执行日、标签日均有明确定义；Promotion Gate 要求全量回测、OOS 验证、成本复核，防止过拟合投产。
- **模块化方向正确**：`src/pipeline/`、`src/reporting/`、`src/research/` 已初步完成 CLI 与算法的分离，`scripts/` 保持薄层入口的意图清晰。
- **数据质量意识强**：`data_quality.py`、`akshare_resilience.py`、`disclosure_calendar.py` 均有单独模块，覆盖率检查、缺失标记、PIT 可用性均有处理。
- **回测口径严谨**：`tplus1_open` 执行、涨停买入失败处理（idle/redistribute）、交易成本三档（10/30/50 bps）均已实现。

### 核心问题概述

| 类别 | 严重度 | 问题简述 |
|------|--------|---------|
| 架构 | 高 | scripts/ 仍大（1000+ 行）且测试直接导入 scripts.*，迁移不完整 |
| 数据 | 高 | 单一数据源（AkShare），无备用、无数据版本控制 |
| 特征 | 中 | 无统一 FeatureRegistry，IC 衰减只告警不降权 |
| 模型 | 中 | M6 LTR 只剩 1 个活跃模型，HPO/Stacking 均默认关闭 |
| 回测 | 中 | 月频锁死，无分布式/并行回测，涨停 redistribute 前视风险未验证 |
| 治理 | 高 | Promoted Registry 实为空，无 OOS 自动监控 |
| 工程 | 高 | 无 CI/CD，无 mypy，测试覆盖率目标仅 70% |
| 可观测性 | 中 | 日志非结构化，无仪表盘，IC 告警无自动关闭机制 |

## 2. 架构与模块化

### 2.1 scripts/ 迁移不完整

**问题**：`scripts/run_monthly_selection_*.py` 各文件仍有 1000–1449 行，README 描述的"核心在 src/pipeline/"仍是进行时而非完成时。更严重的是，测试层直接导入 `scripts.*` 中的函数：

```python
# tests/test_monthly_selection_baselines.py
from scripts.run_monthly_selection_baselines import (...)

# tests/test_config_merge.py
from scripts.run_backtest_eval import DEFAULT_CONFIG, _deep_merge, load_config

# tests/test_fetch_newdata_cli.py
from scripts.fetch_fund_flow import parse_args as parse_fund_flow_args
```

这意味着：（a）这些函数实际上属于 `src/`，却仍停留在脚本层；（b）脚本被当成库使用，测试与脚本形成强耦合；（c）脚本内的任何重构都会破坏测试。

**改进建议**：
- **短期**：在 `src/` 中建立对应模块（如 `src/cli/fetch_fund_flow.py`），将参数解析与入口函数迁入，测试改为导入 `src.*`。
- **中期**：制定"脚本薄层"标准：scripts/*.py 最多 100 行，只做参数解析、路径解析、调用 `src/` 入口函数、落盘报告四件事。可用 `ruff` 规则限制脚本行数。
- **验证**：在 CI 中断言 `scripts/` 中每个文件不超过 150 行（允许少量注释和 CLI 定义）。

### 2.2 pyproject.toml 包发现配置错误

**问题**：

```toml
[tool.setuptools.packages.find]
where = ["src"]
```

项目中实际包路径为 `src/data_fetcher`、`src/pipeline` 等（顶层是 `src`），`where = ["src"]` 会让 setuptools 在 `src/` 目录下寻找顶级包（`data_fetcher`、`pipeline` 等），导致 `import src.pipeline` 失败而 `import pipeline` 可能意外成功，造成混淆。

**改进建议**：

```toml
[tool.setuptools.packages.find]
where = ["."]
include = ["src*"]
```

或使用 namespace package 方式，将 `src` 作为顶级命名空间明确注册。同时在 CI 中加入 `python -c "from src.pipeline.monthly_dataset import build_monthly_selection_dataset"` 冒烟测试。

### 2.3 缺乏内部包导入规范

**问题**：`backtest_runner.py` 中使用绝对路径 `from src.features.fund_flow_factors import ...`，而其他模块使用相对导入 `from ..settings import load_config`，风格不统一，且绝对路径依赖 sys.path 设置正确。

**改进建议**：统一使用相对导入（`from ..features.xxx import yyy`），在 `ruff.lint.select` 中加入 `TID252`（禁止相对到绝对的混用），并在 `pyproject.toml` 中配置 `tool.ruff.lint.isort`。

### 2.4 缺乏类型注解完整性

**问题**：虽然所有文件都有 `from __future__ import annotations`，但没有 `mypy` 配置（`pyproject.toml` 中无 `[tool.mypy]` 节），`Any` 类型在多处使用而无类型守卫，`backtest_runner.py` 中大量字典返回类型为 `Dict[str, Any]`。

**改进建议**：
- 在 `pyproject.toml` 中添加：
  ```toml
  [tool.mypy]
  python_version = "3.10"
  warn_return_any = true
  disallow_untyped_defs = true
  ignore_missing_imports = true
  ```
- 从核心模块（`src/pipeline/monthly_dataset.py`、`src/backtest/engine.py`）开始逐步消灭 `Any`，用 `TypedDict` 替代无类型字典返回值。

## 3. 数据层

### 3.1 单一数据源依赖（AkShare）

**问题**：整个项目依赖 AkShare 作为唯一数据来源。AkShare 是非官方数据接口，存在以下风险：
- API 不稳定，频繁改版（已有 `akshare_resilience.py` 缓解但治标）
- 数据质量依赖第三方清洗，无 tick 级验证
- 基本面数据（`fundamental_client.py`）和资金流数据（`fund_flow_client.py`）覆盖率参差不齐

**改进建议**：
- 定义 `DataSourceAdapter` 抽象接口，将 AkShare 封装为一个实现；预留同花顺（TuShare Pro）或 Wind 适配器接口。
- 将数据源配置化：
  ```yaml
  data_sources:
    daily: akshare          # akshare | tushare | wind
    fundamental: akshare
    fund_flow: akshare
  ```
- 对于关键时间窗口（如每月换仓前后 3 天），增加双源交叉验证逻辑，发现异常时触发人工确认流程。

### 3.2 数据版本控制缺失

**问题**：DuckDB 文件直接就地更新，无 schema migration 机制，无数据快照版本控制。`PREPARED_FACTORS_SCHEMA_VERSION = 2` 仅用于缓存失效判断，不支持 schema 演进和回滚。当表结构变化时（如新增列），旧 DuckDB 文件直接不兼容，需手动重建。

**改进建议**：
- 引入轻量 migration 机制：
  ```python
  # src/data_fetcher/migrations.py
  MIGRATIONS = {
      1: "ALTER TABLE a_share_daily ADD COLUMN IF NOT EXISTS turnover DOUBLE",
      2: "ALTER TABLE a_share_daily ADD COLUMN IF NOT EXISTS pct_chg DOUBLE",
  }
  ```
  在 `DuckDBManager.__init__` 中自动检测并执行未完成的 migration。
- 在 DuckDB 中维护 `schema_version` 元数据表，每次启动时校验并记录版本。
- 为月度关键截面数据（信号日 Parquet 文件）生成内容哈希并在 manifest 中记录，支持数据血缘追踪。

### 3.3 normalize_daily_frame 中的 symbol 提取风险

**问题**：

```python
df["symbol"] = (
    df["symbol"].astype(str).str.extract(r"(\d{1,6})", expand=False)
    .fillna("").str.zfill(6)
)
```

正则 `\d{1,6}` 会匹配任意 1-6 位数字，若 symbol 字段含混合格式（如 "SH600001" 或 "600001.SH"），提取结果可能正确；但若出现 "60000X"、空字符串或非数字，`.fillna("").str.zfill(6)` 会将空字符串填成 "000000"，可能引入幽灵标的，且不会报错。

**改进建议**：
- 改用严格提取：
  ```python
  extracted = df["symbol"].astype(str).str.extract(r"(\d{6})", expand=False)
  invalid_mask = extracted.isna()
  if invalid_mask.any():
      _LOG.warning("发现 %d 行无法提取6位symbol，已丢弃", invalid_mask.sum())
  df = df[~invalid_mask].copy()
  df["symbol"] = extracted[~invalid_mask]
  ```
- 在 `data_quality.py` 的 `validate_daily_frame` 中增加 symbol 格式校验规则。

### 3.4 DuckDB 并发安全边界不清

**问题**：`DuckDBManager` 使用 `concurrent.futures.ThreadPoolExecutor`（`_fetch_workers` 默认 4）并发拉取数据，但只有一个 `self._conn`。DuckDB 在同一进程内可以多线程读，但写入需要单线程或显式加锁。当前代码中未见对 `self._conn` 写操作的显式加锁。

**改进建议**：
- 在 `DuckDBManager` 中添加 `threading.Lock`，所有写操作（`INSERT`、`UPDATE`）使用 `with self._write_lock:` 保护。
- 或者改为"并发拉取、顺序写入"模式：fetch 阶段多线程，完成后将结果批量交给单线程写入。
- 在测试中添加并发写入压力测试用例（`test_concurrent_write.py`）。

## 4. 特征工程

### 4.1 缺乏统一 FeatureRegistry

**问题**：因子计算逻辑分散在 `fundamental_factors.py`、`fund_flow_factors.py`、`shareholder_factors.py`、`backtest_runner.py`（直接内联计算），各模块有自己的特征列常量（`FUNDAMENTAL_RAW_FEATURES`、`FUND_FLOW_RAW_FEATURES`、`SHAREHOLDER_RAW_FEATURES`、`PRICE_VOLUME_FEATURES`）。当需要添加新因子或修改列名时，需同时修改多处常量。

**改进建议**：
- 在 `src/features/` 下建立 `registry.py`：
  ```python
  @dataclass
  class FactorSpec:
      name: str
      family: str       # price_volume / fundamental / fund_flow / shareholder
      compute_fn: Callable
      feature_col: str  # 输出列名
      requires_pit: bool = False
      min_coverage: float = 0.5
      ic_decay_threshold: float = 0.02
  
  FACTOR_REGISTRY: dict[str, FactorSpec] = {}
  
  def register_factor(spec: FactorSpec):
      FACTOR_REGISTRY[spec.name] = spec
  ```
- 统一从 Registry 驱动特征列名，消除散落的字符串常量。
- Registry 提供 `get_active_factors(family)` 方法，IC Monitor 衰减后可自动在 Registry 中标记为 `inactive`。

### 4.2 IC 衰减告警无自动响应机制

**问题**：`ICMonitor.check_decay_alerts()` 检测到 IC 均值跌破阈值后仅触发 Webhook 告警，没有自动降权或关闭因子的机制。人工收到告警后需手动修改 `config.yaml` 中的因子权重，链路断开。

**改进建议**：
- 在 `ICMonitor` 中增加 `get_decayed_factors(window, threshold)` 方法，返回已衰减因子列表。
- 在月度数据集构建阶段（`monthly_multisource.py` 的 `attach_enabled_families`）中调用此方法，自动将衰减因子标记为 `coverage_flag = "ic_decayed"` 并从训练特征中排除。
- 在月度报告中增加"因子衰减状态"表格，显示各因子近 3/6/12 个月滚动 IC 均值和当前状态（active/decayed/watchlist）。

### 4.3 特征覆盖率检查未集成到主链路

**问题**：覆盖率检查主要在 M5（`monthly_multisource.py` 中的 `filter_low_coverage_cols`）执行，M7 报告生成阶段（`monthly_report.py`）对特征覆盖率的依赖通过 `coverage_flag` 隐式传递。若基本面数据更新延迟（常见于季度报告发布窗口），当月覆盖率可能低至 20–30%，但无主动拦截机制。

**改进建议**：
- 在 `MonthlySelectionConfig` 中增加 `min_feature_coverage: float = 0.60`，在 M2 候选池构建阶段即计算每个特征家族的当月覆盖率，低于阈值时发出警告并在报告头部标注"数据质量降级"。
- 建立"特征可用性日历"：记录历史各特征家族的覆盖率时间序列，用于 walk-forward 训练时动态决定是否引入该特征家族。

## 5. 模型层

### 5.1 M6 LTR 模型单一化

**问题**：`M6RunConfig.ltr_models` 当前只有一个活跃条目：

```python
ltr_models: tuple[str, ...] = (
    "xgboost_rank_ndcg",
    # "xgboost_rank_pairwise",  # DEPRECATED P1-4
    # "top20_calibrated",       # DEPRECATED P1-4
    # "ranker_top20_ensemble",  # DEPRECATED P1-4
    # "stacking_ensemble",      # 默认关闭
)
```

单一模型意味着 M6 对 XGBoost Ranker 的超参数非常敏感，且无集成效果。Pairwise 和 Calibrated 被 deprecated 的原因是 after-cost excess 为负，但这可能是超参数问题而非模型结构问题。

**改进建议**：
- **短期**：启用 `hpo_enabled=True`（至少在研究环境），通过 Optuna 对 `xgboost_rank_ndcg` 的 `max_depth`、`learning_rate`、`subsample`、`colsample_bytree`、`min_child_weight` 进行时序感知调优（已有 `hpo_utils.py` 框架）。
- **中期**：引入 LightGBM Ranker 作为第二模型（已在 requirements 可安装），与 XGBoost 进行 Rank IC 比较；两者集成时用简单平均而非 Stacking，降低过拟合风险。
- **Stacking**：`stacking_enabled` 的 OOF 元学习器已有框架（P2-3），建议在 M12 阶段正式验证，验证通过后作为可选 ablation 而非默认路径。

### 5.2 时序模型未集成主链路

**问题**：`src/models/timeseries/` 下有 `lstm_tcn.py`、`ohlcv_norm.py`、`train.py`，实现了 LSTM+TCN 序列模型，但未在任何 `scripts/run_monthly_selection_*.py` 中使用，也没有对应测试或报告。这部分代码处于"死代码"状态，维护成本高于收益。

**改进建议**：
- **短期**：在 `pyproject.toml` 中的 `coverage.run.omit` 中明确排除 `src/models/timeseries/`，或在文件头注明 `# STATUS: experimental, not integrated`。
- **中期**：若要激活，在 M6 LTR 框架中增加 `ltr_model = "tcn_rank"` 选项，将 TCN 序列嵌入作为额外特征输入 XGBoost，而非作为独立打分模型，降低集成复杂度。

### 5.3 HPO 搜索空间与时序感知缺失

**问题**：`hpo_utils.py` 实现了基于 Optuna 的 HPO，但 `hpo_cv_folds = 3` 使用的是 K-Fold 交叉验证而非时序交叉验证，存在未来数据泄露风险（在时序数据上使用非时序 CV 是典型偷看未来的漏洞）。

**改进建议**：
- 将 HPO 的交叉验证改为 `TimeSeriesSplit`，或使用 walk-forward 最后 N 个月作为验证集（purist holdout）：
  ```python
  from sklearn.model_selection import TimeSeriesSplit
  tscv = TimeSeriesSplit(n_splits=hpo_cv_folds, gap=1)
  ```
- 在 `M6RunConfig` 中增加 `hpo_cv_type: str = "timeseries"` 参数，明确区分时序 CV 和非时序 CV 两种模式。
- HPO 结果应写入 manifest（已有框架），并在报告中显示最优超参数和各 fold 的 IC 分布。

## 6. 回测与评估

### 6.1 换仓频率硬编码为月度

**问题**：`BacktestConfig` 和整个 `pipeline/` 的设计隐含月频换仓（`rebalance_rule = "M"`），没有周频、双月、季频的扩展点。在某些市场状态（如高波动期）下，月频换仓过慢；在低 alpha 期，频繁换仓成本吃光超额。

**改进建议**：
- 在 `BacktestConfig` 中将 `rebalance_rule` 参数化，支持 `"W"`（周）、`"M"`（月）、`"BM"`（双月）、`"Q"`（季）。
- 在 `backtest_runner.py` 中将信号日选择逻辑抽象为 `select_rebalance_dates(dates, rule)` 函数（已有 `select_month_end_signal_dates`，需泛化）。
- 在 M10 成本分析阶段，增加"换仓频率 vs 净超额"的敏感性图，帮助选择最优换仓频率。

### 6.2 涨停 redistribute 模式前视偏差未验证

**问题**：`limit_up_mode = "redistribute"` 将涨停票的新增权重"均匀分配给同日其他可买标的"。但"同日其他可买标的"在实际执行时需要知道哪些票没有涨停，这依赖当日开盘价信息——而确认当日开盘涨停是否发生，理论上需要等开盘后几分钟。在极端情况下，redistribute 逻辑可能依赖实际不可知的信息。

**改进建议**：
- 在 `BacktestResult.meta` 中增加 `limit_up_redistribute_events` 统计，记录每个换仓日有多少票触发 redistribute、分配了多少权重。
- 增加专项回测：在 `redistribute` 模式下，将被分配权重的标的的 T+1 收益与等权基准比较，验证是否存在系统性偏差。
- 在报告中明确标注 `redistribute` 结果的置信区间，并与 `idle` 模式的对照结果并排展示。

### 6.3 Oracle 结论过于乐观未做充分说明

**问题**：`docs/plan.md` 显示 U1 Top20 Oracle 月均超额约 74%，这个数字远超任何实际策略。Oracle 的意义是"上限诊断"，但报告中没有明确说明 Oracle 和实际策略之间的 overlap 极低（plan.md 提及"当前特征离 oracle imitation 远"），可能误导读者。

**改进建议**：
- 在月度报告中增加"Oracle Gap 分析"节，显示：（a）当月 Oracle Top20 与策略 Top20 的 overlap 数量；（b）策略 Rank IC 与 Oracle 完美 Rank IC（=1.0）之间的差距；（c）"可实现超额 = oracle × IC²"的粗略估计。
- Oracle 报告单独归档，不与实际策略报告合并展示，避免视觉上的误导。

### 6.4 walk-forward 的 query group 构建细节缺乏文档

**问题**：M6 LTR 使用 walk-forward XGBoost Ranker，XGBoost Ranker 需要 `qid`（query group id）来定义每个排序组。当前代码中 `qid` 的构建方式（按月截面分组）没有文档说明，且不同月份的截面大小差异可能导致 NDCG 计算不稳定。

**改进建议**：
- 在 `monthly_ltr.py` 中为 `build_ltr_relevance` 和 query group 构建添加详细注释，说明 qid 对应月份、每组期望大小（U1 候选池通常 200–500 只）及 NDCG@K 的 K 值选择依据。
- 在 M6 报告中增加"每月截面样本量分布"直方图，确认各月截面大小稳定，避免小截面月份对 NDCG 均值产生过大影响。

## 7. 研究治理

### 7.1 Promoted Registry 实为空

**问题**：`configs/promoted/promoted_registry.json` 虽然有内容，但 `plan.md` 明确指出"生产状态：无研究候选进入生产"。仓库中有两个 promoted config JSON 文件（`monthly_selection_u1_top20_indcap3_hardcap_baseline.json`、`monthly_selection_u1_top20_m8_natural_soft_gamma0_20.json`），但均未被激活到生产。这意味着整个系统运行两年多（历史报告追溯至 2023 年），没有任何策略达到 promotion 标准。

**这可能是合理的研究保守性，也可能是 Promotion Gate 阈值设置过高**。

**改进建议**：
- 明确量化 Promotion Gate 通过标准（如：年化超额 > X bps，Sharpe > Y，最大回撤 < Z，样本外 Rank IC > W）并写入 `configs/promoted/README.md`。
- 对"hard-cap baseline"（当前 after-cost 月均超额约 0.022）进行全量 gate 评估：它是否已经满足所有 gate 条件？若满足，应当 promote；若不满足，需明确哪个条件未达标。
- 建立"研究候选漏斗"文档，跟踪每个候选从 M5 进入到 promotion 或拒绝的完整决策记录。

### 7.2 OOS 监控无自动化

**问题**：策略 promotion 后，没有自动化的样本外（OOS）表现监控机制。每月需人工运行报告脚本并与基准比较，且当前连 promotion 都没有，OOS 监控更是空白。

**改进建议**：
- 建立 `src/monitoring/oos_tracker.py`：每月换仓后自动记录当月实际 Top20 的持有收益，与历史回测预测超额对比，计算"预测-实现超额差异"的滚动统计。
- 设定 OOS 降级触发条件（如连续 3 个月实现超额低于回测均值的 50%），触发时自动发出告警并启动重新评估流程。
- 在 `promoted_registry.json` 中为每个 promoted 配置添加 `oos_tracking_start_date` 和 `oos_performance_summary` 字段。

### 7.3 配置 Schema 缺少验证

**问题**：`load_config()` 只做 `yaml.safe_load()` 后直接返回 `Dict[str, Any]`，没有 Schema 验证。错误的配置键（如拼写错误 `top_kk = 20`）会静默失败，导致系统使用代码内默认值而用户毫不知情。

**改进建议**：
- 引入轻量 Schema 验证（使用 `pydantic` 或 `jsonschema`）：
  ```python
  class PortfolioConfig(BaseModel):
      top_k: int = Field(ge=5, le=100, default=20)
      weight_method: Literal["equal", "fractional", "risk_parity"] = "equal"
      max_single_weight: float = Field(ge=0.01, le=0.20, default=0.10)
  ```
- 在 `DuckDBManager.__init__` 和各 `RunConfig` dataclass 的构建处加入配置校验，配置错误时立即抛出带详细说明的 `ValueError`。
- 提供 `python scripts/env_check.py --validate-config config.yaml` 命令，专门做配置有效性检查。

## 8. 工程实践

### 8.1 无 CI/CD

**问题**：仓库中没有 `.github/workflows/` 目录（已确认），也没有其他 CI 配置（GitLab CI、Jenkins 等）。代码变更、脚本重构、配置更新后没有自动化测试门禁，依赖手动运行 `pytest`。

**改进建议**：建立 GitHub Actions 工作流 `.github/workflows/ci.yml`：

```yaml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: {python-version: "3.10"}
      - run: pip install -e ".[dev]"
      - run: ruff check src/ scripts/ tests/
      - run: mypy src/ --ignore-missing-imports
      - run: pytest --cov=src --cov-fail-under=75
      - name: Smoke test imports
        run: python -c "from src.pipeline.monthly_dataset import build_monthly_selection_dataset"
```

### 8.2 测试覆盖率目标过低且未强制执行

**问题**：`pyproject.toml` 设置 `--cov-fail-under=70`，核心 pipeline 期望 ≥ 80%，但没有 per-module 的覆盖率门禁。当前测试集（26 个测试文件）中大量测试依赖 mock 数据，对真实数据路径覆盖不足。

**改进建议**：
- 将整体覆盖率目标提升至 75%，并为核心模块单独设置：
  ```toml
  [tool.coverage.report]
  fail_under = 75
  
  # 分模块检查可在 CI 中通过 coverage report --include="src/pipeline/*" 实现
  ```
- 增加以下高价值测试用例：
  - `test_data_quality.py`：构造含 OHLC 逻辑错误、日期跳空的 DataFrame，验证 `run_quality_checks` 正确报告。
  - `test_backtest_engine.py`：构造含涨停日的简单案例，验证 `idle` 和 `redistribute` 两种模式的收益差异符合预期。
  - `test_ic_monitor.py`：模拟 IC 衰减序列，验证告警触发时机和 `get_decayed_factors` 返回结果。

### 8.3 日志系统非结构化

**问题**：`src/logging_config.py` 使用标准文本格式日志，输出到 `data/logs/`。在需要诊断历史问题时（如"上个月换仓时哪些票触发了涨停过滤？"），需要手动 grep 文本日志，效率低下。

**改进建议**：
- 将关键事件日志改为结构化 JSON 格式（可用 `python-json-logger`）：
  ```python
  import json
  _LOG.info("涨停过滤", extra={
      "event": "limit_up_filter",
      "symbol": sym,
      "signal_date": str(sig_date),
      "action": "removed_from_pool"
  })
  ```
- 在 DuckDB 中建立 `run_events` 事件日志表，存储关键决策事件，支持后续 SQL 查询分析。
- 提供 `scripts/query_run_events.py --event-type limit_up_filter --month 2026-04` 查询工具。

### 8.4 缺乏性能基准测试

**问题**：项目在全市场 5000+ 股票上运行月度因子计算和回测，单次全量运行时间可能超过 30 分钟（含数据拉取）。但没有性能基准记录，无法判断代码变更是否引入性能退化。

**改进建议**：
- 使用 `pytest-benchmark` 为核心计算步骤建立基准：
  ```python
  def test_factor_compute_benchmark(benchmark):
      benchmark(compute_all_factors, sample_daily_df)
  ```
- 在 `scripts/env_check.py` 中增加计时测试：用固定规模的合成数据测试因子计算、回测引擎的执行时间，并与基准值对比。

## 9. 可观测性与运营

### 9.1 无监控仪表盘

**问题**：IC Monitor 有 DuckDB 存储，但无可视化界面。每月研究结果是 Markdown 文件，无法快速比较多策略的历史超额趋势、因子 IC 走势、行业暴露变化。

**改进建议**：
- **短期**：在 `scripts/` 中增加 `generate_dashboard.py`，输出一个自包含的 HTML 报告（使用 `plotly` 或 `altair`），包含：（a）各月超额 bar chart；（b）因子滚动 IC 折线图；（c）候选池大小历史；（d）行业集中度热力图。
- **中期**：若团队接受第三方服务，可将 DuckDB 数据推送至 Grafana（配合 DuckDB Grafana plugin）或 Metabase，实现实时监控。

### 9.2 告警渠道单一且与业务耦合

**问题**：`src/notify/__init__.py` 中的 `WecomWebhookHandler` 与企业微信硬绑定，且 URL 需要直接传入（敏感信息）。IC 衰减告警是唯一使用通知系统的场景，其他运营事件（数据拉取失败、配置校验失败、月度报告生成完成）均无通知。

**改进建议**：
- 将 Webhook URL 移入配置文件：
  ```yaml
  notify:
    wecom_webhook_url: ${WECOM_WEBHOOK_URL}  # 环境变量注入
    enabled_events: [ic_decay, data_fetch_fail, report_ready]
  ```
- 定义 `NotifyEvent` 枚举和 `EventBus` 单例，各模块通过 `event_bus.emit(NotifyEvent.DATA_FETCH_FAILED, ...)` 发布事件，解耦业务逻辑与通知渠道。
- 在 `scripts/run_pipeline_all_steps.sh` 完成后自动发送"月度报告已就绪"通知，包含关键指标摘要。
