## 总体评估

项目研究纪律严谨，PIT 安全性和评估口径已经相当成熟。主要不足集中在六个维度：**工程债**（scripts 过重、测试覆盖不足）、**数据层瓶颈**（单源依赖、关键数据未接入）、**模型天花板**（Rank IC ≈ 0.10，距 Oracle 上限差距巨大）、**回测统计严谨性**（缺少显著性检验）、**组合执行层**（冲击成本建模粗糙）和**运维自动化**（全靠手工驱动）。

## 一、工程质量与代码架构

### 1.1 脚本层过重（高优先级）

**问题**：`scripts/` 下最大脚本达 1384 行（`run_monthly_selection_m8_natural_industry_constraints.py`），多数脚本在 500–1200 行之间。README 明确"核心逻辑应在 `src/`"，但实际仍有大量算法逻辑残留脚本层。两处存在已知"本地函数副本"，文档承认不应作为后续维护主位置，但并未限期清理。

**风险**：同一逻辑在脚本层和 `src/` 层出现分叉，修改一处时另一处悄然落后，且测试只覆盖脚本层副本时会给出虚假的安全感。

**改进建议**：
- 聚焦瘦身最大的 3 个脚本（ltr 700 行、m8 757 行、report 511 行），将核心逻辑迁入 `src/pipeline/`，目标 < 250 行。baselines(223) 和 multisource(314) 已部分瘦身，当前状态可接受。
- 共享基础设施（`cli_helpers` + `research_runner`）已就位，后续脚本直接复用即可消除 ~80 行/脚本的重复样板。
- CI 中已有 `>200` 行 warning 检查，作为软性提醒不阻断。

### 1.2 测试覆盖率与模块盲区（高优先级）

**问题**：CI 覆盖率门槛仅 40%，但 `pyproject.toml` 注释期望核心 pipeline 模块 ≥ 80%。更严重的是，以下 `src/` 子模块**完全没有对应的 `tests/test_*.py` 文件**：

| 无测试的模块 | 风险 |
|---|---|
| `src/analysis/` | 容量报告、rebalance 敏感性分析是 M10 核心输出，无验证 |
| `src/backtest/` | 回测引擎核心，涨停处理逻辑复杂，只有集成测试 |
| `src/monitoring/oos_tracker.py` | OOS 追踪 SQL 写入/查询逻辑，无单元测试 |
| `src/notify/` | Webhook 发送逻辑，无 mock 测试 |
| `src/portfolio/` | 协方差估计、最优化约束，无单元测试 |
| `src/market/regime.py` | Regime 分类逻辑，只有 1 个 smoke test |
| `src/models/timeseries/` | LSTM/TCN 代码无测试（且疑似废弃） |

**改进建议**：
- 将 CI 覆盖率门槛分两步提升：当前 55%（已达标），下一步到 58-60%。
- 优先为 `portfolio/optimizer.py`（62%，91 行未覆盖）和 `portfolio/covariance.py`（65%，85 行未覆盖）补单元测试——这是生产组合层的核心代码。
- `src/monitoring/oos_tracker.py` 使用 in-memory DuckDB（`:memory:`）做单元测试，不依赖真实文件。
- 70% 目标下调：剩余 15 个百分点集中在 `backtest_runner.py`（586 行未覆盖，14%）和已废弃的 `timeseries/`（410 行），投入产出比低。

### 1.3 静态类型覆盖不足（中优先级）

**问题**：`mypy` 在 CI 中以 `|| true` 运行，失败不阻断。仅 3 个模块强制 `disallow_untyped_defs`（`monthly_dataset`、`ic_monitor`、`db_manager`），其余大量函数无类型注解。`src/pipeline/monthly_multisource.py`、`src/pipeline/backtest_runner.py`（56176 字节）等高复杂度模块均无约束。

**改进建议**：
- 去掉 `|| true`，改为 `mypy ... --no-error-summary || exit 1`，至少让报告可见并记录在 CI artifact。
- 将强制模块白名单扩展到 `src/backtest/`、`src/portfolio/`、`src/features/`，以季度为单位渐进覆盖。
- 对超过 500 行的函数（如 `backtest_runner.py` 中的大函数）补充 `-> ReturnType` 注解，作为下次重构的入口。

### 1.4 废弃代码未清理（低优先级）

**问题**：`src/models/timeseries/`（`lstm_tcn.py`、`train.py`，合计约 28K 字节）仍在代码库中，但研究停止清单第 10 条明确"不把深度学习作为当前第一主线"。`notebooks/` 目录只有 `.gitkeep`，说明研究探索全靠脚本而非 notebook，且 LSTM 相关代码无活跃调用链。

**改进建议**：
- 将 `src/models/timeseries/` 移入 `archive/` 分支或在 `__init__.py` 中加 `raise DeprecationWarning`，明确告知调用者。
- 若确认未来 M11+ 不接入时序深度模型，直接删除此目录并在 `docs/` 留下一条决策记录。

## 二、数据层

### 2.1 单一数据源风险（高优先级）

**问题**：全部市场数据依赖 AkShare 一个来源。`scripts/akshare_network_doctor.py` 的存在说明网络不稳是常态，而一旦 AkShare API 格式变化（历史上曾多次改变），整个管线会立刻失效。`akshare_resilience.py` 提供了重试和本地缓存回退，但没有"备用数据源"层。

**改进建议**：
- 在 `src/data_fetcher/adapter.py` 中增加 `DataSourceAdapter` 抽象层，让 `DuckDBManager` 通过接口而非直接调用 AkShare 函数获取数据。
- 备选数据源（即便只是 Tushare 作为降级）通过同一接口注入，发生 AkShare 网络故障时可自动切换。
- 为 `adapter.py` 写单元测试，mock 底层数据源调用。

### 2.2 fund_flow 历史太短（高优先级）

**问题**：`FAMILY_DATA_START["fund_flow"] = "2023-10-01"`，只有约 2.5 年历史（≈30 个月）。Walk-forward 训练时资金流因子的有效训练折叠数极少，统计功效弱，且覆盖的市场状态（2023 下半年至今）以震荡为主，缺乏完整牛熊周期验证。

**改进建议**：
- 探索 AkShare 中是否有更早日期的资金流数据（部分指标可追溯到 2020 年）；若有，优先回填。
- 在 M5 增量 delta 报告中，明确标注"资金流因子训练集仅 N 个月"作为不确定性来源，避免高估其泛化性。
- 将资金流因子的 `min_coverage` 门槛提高，当前折叠样本 < 18 个月时自动降级为"辅助特征"而非主特征。

### 2.3 M11 新数据仍未启动（中优先级）

**问题**：M11 优先数据清单（北向资金、主题/概念 breadth、融资融券、龙虎榜）已在计划中列出，但状态是"待启动"，且 M10 还未完成。这意味着当前模型的特征集（价量 + 行业宽度 + 基本面 + 资金流 + 股东）已经跑了若干轮迭代而没有引入真正新的信息来源。

**改进建议**：
- M10 压力测试期间（不影响主线）可同步启动北向资金的数据接入和质量诊断（只做 M1：数据质量 + PIT 检验，不进模型）。
- 为每个新数据族建立"数据接入 checklist"模板，包括：PIT 安全性验证、覆盖率报告、与现有因子相关性矩阵、历史长度确认，避免重复发明数据接入流程。

### 2.4 股票名称 UNKNOWN 问题（低优先级，已知）

**问题**：M7 推荐报告中股票名称字段显示 UNKNOWN，计划中已知但尚未修复（计划优先级标注为下一步）。这是一个影响报告可读性的 UX 问题。

**改进建议**：
- `scripts/fetch_stock_names.py` 已存在，将其集成进 `run_monthly_selection_report.py` 的前置步骤，而非手工单独运行。
- 在 CI smoke test 中加入名称覆盖率断言：`top20_name_unknown_rate < 0.1`。

## 三、模型层

### 3.1 模型能力天花板评估（高优先级）

**问题**：当前最优模型（M5 ExtraTrees，U1，Top20）after-cost 月均超额约 1.14%，Rank IC 约 0.107。Oracle Top20 月均超额约 74.2%（这是完美预见未来的上限）。两者的差距反映的不只是模型能力问题，主要是"可用特征集本身对 Oracle 的解释力弱"——Oracle overlap 检验也明确证实了这一点。

**改进建议**：
- 在每次新特征族接入时，优先计算该特征族对 Oracle Top-K 的"条件 IC"（即在 Oracle 样本上的 IC），而不只是全截面 IC。这样能更直接地度量新特征对选择力的贡献。
- 将"Oracle 条件命中率"作为正式 M11 新数据族的接入 gate，要求新数据至少提升 Oracle overlap 2 个百分点以上，才允许进入模型主线。

### 3.2 M6 LTR 诊断不充分（中优先级）

**问题**：M6 XGBoost rank NDCG 在 U1 池上完全失败（after-cost 超额 0.000214，接近零），只在 U2+Top20 上有弱证据（超额 1.21% 但 Rank IC 极低 0.007）。计划将其标注为"watchlist/sleeve"，但没有清晰的分析说明为什么 U2 能通过而 U1 失败，也没有探索 NDCG 超参（`ndcg_exp_gain`、列采样）对结果的敏感性。

**改进建议**：
- 做一次 M6 vs M5 的失败归因分析：逐年比较 U1 和 U2 的候选池构成差异，找出 U1 中导致 M6 失败的具体股票群体（是 ST 残余？微盘？涨跌停候选？）。
- 若归因结论是"U1 噪声过多"，则优化 U1 过滤规则，而非放弃 LTR 方法本身。

### 3.3 Regime Switch 参数未做稳健性检验（中优先级）

**问题**：`RegimeConfig` 中的 bull/bear 阈值（`bull_return_threshold=0.05`、`bear_return_threshold=0.04`）和各状态下的乘数都是硬编码的启发式数值，没有参数敏感性报告。在 A 股 2018–2026 跨越了多个牛熊周期，Regime 分类的边界条件对组合收益影响可能是非线性的。

**改进建议**：
- 添加一个 `regime_sensitivity_grid` 脚本，在 `bull_return_threshold ∈ [0.03, 0.05, 0.07]` × `bear_return_threshold ∈ [0.03, 0.04, 0.05]` 网格上运行 M8 baseline，报告 Sharpe 和超额的分布范围，量化 Regime 参数的风险暴露。
- 将此敏感性报告作为 M12 Promotion Package 的必要附件。

### 3.4 缺少统计显著性检验（中优先级）

**问题**：当前评估只报告 Rank IC 均值、after-cost 超额均值、分桶 spread 等点估计。没有 t-test、bootstrap 置信区间或 Fama-MacBeth 回归，无法判断历史 Rank IC ≈ 0.10 是否显著非零（考虑到截面相关性和时序自相关，标准 t-test 会高估显著性）。

**改进建议**：
- 在 M5/M8 评估报告中加入：
  - Rank IC 的 Newey-West 调整 t-统计量（lag=6）
  - 月度超额的 bootstrap 95% 置信区间（1000 次重采样，保留时序顺序）
  - `Information Ratio = mean(excess) / std(excess)`（按月）
- 将"IR > 0.5 且 NW-adjusted t > 2.0"作为进入 M12 Promotion Package 的统计门槛。

## 四、回测层

### 4.1 冲击成本建模过于简单（高优先级，M10 核心）

**问题**：当前 VWAP 冲击模型使用固定参数（`vwap_slippage_bps_per_side=3.0`，`vwap_impact_bps=8.0`），对所有股票一视同仁。实际上小市值/低成交额股票的冲击成本可以是大市值的 10-50 倍。`analysis/capacity_report.py` 已有按 `amount_20d` 分层的框架，但还没有与回测引擎中的成本模型联通。

**改进建议**：
- 实现分层冲击成本：按 `amount_20d` 将股票分为大/中/小三档，每档配置独立的 `slippage_bps` 和 `impact_bps`，参数参考国内公募量化研究（小市值冲击成本建议 20–50 bps）。
- 回测结果报告中新增"冲击成本来源分解"：哪些月份、哪些持仓贡献了多少冲击，作为 M10 压力测试的核心证据。
- M10 通过条件中"30 bps 后不低于市场等权"应修正为"使用分层成本模型后，不低于市场等权"。

### 4.2 换手分析不充分（中优先级）

**问题**：换手约束只有"half L1 月度换手约束"，但没有分析换手率与超额收益的 tradeoff 曲线（即：降低换手限制后，超额收益在多大程度上依赖高换手？）。若高超额月份集中在高换手月份，则实际可行性存疑。

**改进建议**：
- 在 M10 报告中加入"换手-超额散点图"：x 轴为月度换手率，y 轴为当月超额收益，区分 bull/bear/oscillation 状态，检验超额是否对换手有依赖。
- 计算"换手调整后 IR"：`IR_adj = mean(excess) / (std(excess) + λ * mean(turnover))`，其中 λ 为成本斜率，作为与 hard-cap baseline 比较的补充指标。

### 4.3 缺少过拟合风险量化（中优先级）

**问题**：Walk-forward 回测是时序安全的，但没有以下检验：1）特征重要性跨折叠的稳定性（某折叠中重要特征在下一折叠是否消失）；2）参数网格搜索后的多重检验校正（当前 M8 自然化约束运行了多个 `gamma` 值）；3）持有期收益的自相关检验（若相邻月度超额强正相关，有效样本量被高估）。

**改进建议**：
- 输出特征重要性跨折叠的变异系数（CV）：CV > 1.0 的特征标注为"不稳定特征"，在 M12 Promotion Package 中必须有说明。
- 对 M8 gamma 网格搜索结果应用 Bonferroni 校正，报告"校正后最优 gamma 下超额是否仍显著"。

## 五、组合与执行层

### 5.1 默认等权与生产组合之间的鸿沟（高优先级）

**问题**：研究评价默认使用等权，而 `src/portfolio/` 提供了风险平价、最小方差、均值-方差等多种方法，但这些方法在 M10 之前完全未进入主线评估。"等权"结果能否代表未来实际生产组合的表现是未知的。

**改进建议**：
- 在 M10 压力测试中，将等权 vs 风险平价 vs 最小方差的 Sharpe/超额 side-by-side 报告作为必输出项，而非留到 M12 才做。
- 明确记录生产形态将使用哪种组合方法，以及选择依据。

### 5.2 买入失败处理逻辑的边界案例（中优先级）

**问题**：涨停买入失败的 `redistribute` 模式将权重"均匀分配给同日其他可买标的"，但没有检验以下边界：当 Top20 中有多只同日涨停时，redistribution 后的行业集中度是否仍满足 M8 约束；2021–2022 年涨停潮期间（北交所科创板上市首日），这一比例较高。

**改进建议**：
- 在 M10 买入失败分析中加入"多票同日涨停"的月份统计，报告这些月份 redistribution 后的最大行业占比。
- 若有超过 3 只同日涨停，redistribute 后行业集中度超标时，提供第三种处理选项：`redistribution_with_industry_cap`（在 redistribution 时同时保持行业约束）。

## 六、运维与自动化

### 6.1 月度流程全靠手工驱动（高优先级）

**问题**：月度选股运行没有任何自动化调度。每月需手工依次执行 fetch_only → M2 → M3 → M4 → M5 → M6 → M7 等 7+ 个步骤，且 `scripts/run_pipeline_all_steps.sh` 是否能一键执行尚不清楚（文件只有 4KB，可能只是部分步骤）。手工执行容易因顺序错误或遗漏步骤造成数据断层。

**改进建议**：
- 完善 `scripts/run_pipeline_all_steps.sh`，使其能以 `--config config.yaml` 一键完成 M2→M7 全链路，中途任何步骤失败立即 `exit 1`，并通过 `src/notify/` 发送失败告警。
- 考虑使用 cron 或 GitHub Actions 的 `schedule` 触发器，在每月最后一个交易日后自动运行数据更新步骤（`fetch_only`），确保数据不断层。

### 6.2 OOS Tracker 有实现无数据（中优先级）

**问题**：`src/monitoring/oos_tracker.py` 已完整实现，但由于 promoted registry 为空，从未有真实 OOS 记录被写入。OOS 监控的价值在于"发现策略退化"，而目前连最初几个月的预测-实现差异都没有积累。

**改进建议**：
- 在 M10 完成后，即使正式 promotion 还未发生，也应从"研究候选"层面开始记录 OOS：为 M8 natural 候选（`monthly_selection_u1_top20_m8_natural_soft_gamma0_20`）手工补录 2026-04/05 两个月的预测 vs 实现超额，建立 OOS 基线。
- 将 OOS 记录嵌入 `run_monthly_selection_report.py` 的最后一步：每次生成 M7 报告后，自动将预测超额写入 `oos_tracking` 表，等下月数据可用后再补录实现值。

### 6.3 文档版本管理混乱（低优先级）

**问题**：`docs/` 目录存在 `plan.md`、`plan-05-03.md`、`plan-05-03(1).md`、`plan-05-04.md` 四个并存的计划文档，以及 20+ 个不同日期的中间报告 `.md` 文件（部分达到 69KB+）。主计划文档中声明"唯一主计划（canonical）"，但历史版本散落在同级目录，容易引起混淆。

**改进建议**：
- 将历史计划文档归档到 `docs/archive/plans/`，只保留 `docs/plan.md` 作为活跃文档。
- 在 `docs/README.md` 中维护一张"文档索引表"，注明每个文档的角色（canonical / archived / evidence）。
- 大型中间报告（> 50KB 的 `.md`）迁移到 `docs/reports/` 按月份目录，不与当前计划文档混放。
