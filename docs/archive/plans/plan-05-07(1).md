## 一、缺点汇总

### 1. 🔴 DuckDB 连接未受保护——异常时资源泄漏

**位置**：`src/pipeline/backtest_runner.py:127`（`load_daily_from_duckdb`）

```python
# 现状（有缺陷）
con = duckdb.connect(db_path, read_only=True)
df = con.execute(sql).df()   # 若此行抛异常，下面的 close 不会执行
con.close()
```

`con.execute()` 如果因 SQL 错误或数据问题抛出异常，`con.close()` 永远不会被调用，在批量跑月度回测时会累积大量孤立连接。相比之下，同文件的 `_attach_pit_fundamentals`（第 404 行）已经用了 `try/finally`，但 `load_daily_from_duckdb` 没有。

**类似问题还存在于**：
- `src/features/fund_flow_factors.py`
- `src/features/northbound_factors.py`
- `src/features/northbound_regime_factors.py`
- `src/features/shareholder_factors.py`（两处）
- `src/features/margin_trading_factors.py`
- `src/features/concept_factors.py`
- `src/features/lhb_factors.py`

---

### 2. 🔴 `_winsor_zscore` 重复定义且行为不一致

**位置**：`src/pipeline/monthly_dataset.py:405` 与 `src/pipeline/monthly_multisource.py:180`

两份实现在空值处理上有差异：

| 版本 | 样本不足时返回值 | inf 处理 |
|------|----------------|----------|
| `monthly_dataset.py` | `pd.Series(np.nan, ...)` | 无 |
| `monthly_multisource.py` | `pd.Series(0.0, ...)` | 替换为 nan |

当候选池较小时，两条管线对同一截面会产生不同的 z-score 值，破坏了跨管线比较的可复现性。

---

### 3. 🟠 zscore/标准化实现分散——存在 5+ 份平行实现

| 文件 | 函数名 |
|------|--------|
| `src/pipeline/backtest_runner.py` | `_zscore_clip` |
| `src/pipeline/monthly_dataset.py` | `_winsor_zscore` |
| `src/pipeline/monthly_multisource.py` | `_winsor_zscore`（不同版本） |
| `src/features/standardize.py` | `zscore_cross_section`, `zscore_by_date` |
| `src/models/rank_score.py` | `cross_section_zscore` |

已有一个 `src/features/standardize.py` 的规范实现，但各 pipeline 模块没有复用它，而是各自内联实现，导致小 bug 修复需要同步改多处。

### 4. 🟠 `load_daily_from_duckdb` 与 `read_daily_from_duckdb` 功能重叠

**位置**：`src/pipeline/backtest_runner.py:123` vs `src/pipeline/monthly_dataset.py:97`

两个函数都从 `a_share_daily` 读取日线数据并做基本归一化，但接口不兼容（一个接受连接路径字符串，另一个接受已开好的连接对象）。`backtest_runner.py` 的版本还用 f-string 把日期直接拼入 SQL（见下条），而 `monthly_dataset.py` 已改用参数化查询。

---

### 5. 🟠 部分 SQL 仍用 f-string 拼接动态值

**位置**：`src/pipeline/backtest_runner.py:128-135`

```python
sql = f"""
    SELECT ...
    WHERE trade_date >= '{start_dt}' AND trade_date <= '{end}'
    ...
"""
```

`start_dt` 和 `end` 来自外部 `str` 参数，虽然当前调用路径受控，但 f-string 拼接日期是不良习惯，一旦参数来源变化就会产生 SQL 注入风险。`monthly_dataset.py:read_daily_from_duckdb` 已用 `?` 占位符，标准不统一。

---

### 6. 🟠 `_wilder_atr` 用 Python for 循环——大数据集性能瓶颈

**位置**：`src/pipeline/backtest_runner.py:155-175`

```python
for i in range(period + 1, len(close)):   # 逐行迭代
    prev_atr = atr.iloc[i - 1]
    ...
```

对 5000 只股票 × 500 天的数据，这个循环会成为明显热点。Wilder ATR 可以用 `pandas.Series.ewm(alpha=1/period, adjust=False)` 一行向量化完成。

---

### 7. 🟠 `backtest_runner.py` 职责过重（1342 行）

该文件同时承担：数据加载、技术指标计算（KDJ、ATR、RSI 等）、因子缓存管理、截面打分、IC 加权权重构建、Regime 调权、Top-K 权重优化。每次修改任意一个环节都需要在这 1342 行中定位，且单元测试难以隔离各组件。

---

### 8. 🟠 `FACTOR_REGISTRY` 全局可变字典——线程不安全

**位置**：`src/features/registry.py`

```python
FACTOR_REGISTRY: dict[str, FactorSpec] = {}
```

在多线程批量因子注册场景（如并发回测或多进程管线）中，对这个全局字典的并发读写没有任何锁保护。当前 `db_manager.py` 已使用 `threading.Lock`，但 `FACTOR_REGISTRY` 没有。

### 11. 🟡 Regime 分类使用硬编码阈值，缺乏动态校准

**位置**：`src/market/regime.py`，`RegimeConfig` 默认值

```python
bull_return_threshold: float = 0.05   # 固定
bear_return_threshold: float = 0.04   # 固定
vol_threshold_ann: float = 0.25       # 固定
```

这些阈值基于历史经验设定，但 A 股不同周期的波动率结构差异显著（例如 2020–2021 与 2023–2024 的年化波动率均值差距超过 30%）。当市场结构漂移时，同样的阈值会导致 Regime 分类系统性偏移，影响动态权重调整的有效性，而当前没有任何校准或预警机制。

### 12. 🟡 OOS 监控维度单薄

**位置**：`src/monitoring/oos_tracker.py`

当前只记录和比较"月度超额收益"。判断策略是否退化仅依赖 `realized_excess_monthly < threshold_ratio * predicted_excess_monthly`，缺失：

- 月度 Rank IC 追踪（因子预测能力是否衰减）
- 最大单月亏损预警（尾部风险变化）
- 换手率趋势（是否悄悄提高摩擦成本）
- 持仓集中度漂移（行业暴露是否在 OOS 中悄悄扩大）

#### P1-1：修复所有 DuckDB 连接泄漏

将所有裸 `con = duckdb.connect(...)` + `con.close()` 模式统一改为上下文管理器：

```python
# 改前
```

### P2 — 短期改进（2 周内）

#### P2-1：向量化 `_wilder_atr`

```python
# 改后（基于 ewm）
def _wilder_atr(high, low, close, period=14):
    tr1 = high - low
    tr2 = (high - close.shift(1)).abs()
    tr3 = (low - close.shift(1)).abs()
    tr = pd.concat([tr1, tr2, tr3], axis=1).max(axis=1)
    return tr.ewm(alpha=1.0 / period, min_periods=period, adjust=False).mean()
```

对 5000 只股票 × 500 天的全量回测，预期节省 30–60 秒运行时间。

**验收**：`test_backtest_engine.py` 中增加 ATR 数值对比测试（相对误差 < 1e-6）。

```python
params = [start_dt, end]
```
