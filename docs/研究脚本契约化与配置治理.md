## 0. 当前判断

项目已经不是单纯探索脚本集合，而是形成了清晰的月度选股研究主线：

```text
AkShare / DuckDB 数据
-> 月度 PIT-safe 截面数据集
-> baseline / multisource / LTR / industry constraint 研究脚本
-> leaderboard、gate、topk holdings、report、manifest
-> promoted registry
-> config.yaml.example / config.yaml 生产默认方法
```

当前生产准入边界已经存在：

- `configs/promoted/promoted_registry.json` 是唯一 promotion registry。
- 当前 active promoted 配置是 `monthly_selection_u1_top20_indcap3_hardcap_baseline`。
- `config.yaml.example` 已指向 promoted 默认方法，不应承载未 promotion 的研究候选。
- `configs/backtests/`、`config.yaml.backtest` 继续作为研究配置快照，不直接进入生产。

真正需要治理的不是“缺少框架”，而是：

1. `scripts/run_*.py` 数量多、体量大，很多脚本已经达到 30KB~50KB，`run_backtest_eval.py` 更大。
2. 多个脚本各自拼 `research_config_id`、`output_stem`、manifest、leaderboard、gate、doc 路径，字段语义容易漂移。
3. `src/models/data_slice.py` 已有切片哈希能力，但各研究脚本还没有统一的 `DataSlice` 入口对象。
4. `src/models/experiment.py` 已有本地 JSONL/CSV 实验记录能力，但多数研究脚本仍以各自 manifest 为主，没有统一 `ExperimentResult` 出口。
5. promoted registry 已经管住生产边界，但研究结果到 promotion 证据之间还缺一层统一的、可验证的结果契约。

结论：**保留现有脚本入口，新增轻量契约层；先治理主线月度选股脚本，再治理历史诊断脚本；promoted registry 只收最终 promotion，研究过程用本地 manifest / JSONL 索引承接。**

## 1. 不做什么

本轮计划明确不做以下事情：

| 不做项 | 原因 |
| --- | --- |
| 不引入 MLflow / DVC / Weights & Biases | 项目是本地化研究流水线，当前痛点是契约一致性，不是远端实验平台。 |
| 不把所有 `run_*.py` 改成统一框架 CLI | 研究脚本仍然需要保持快速迭代和可单独运行。 |
| 不删除现有 manifest / report / CSV 产物 | 这些是既有证据链，应该被统一登记，而不是推倒重来。 |
| 不把所有研究候选写进 `promoted_registry.json` | promoted registry 只记录通过完整 gate 的生产候选，研究候选应进入研究结果索引。 |
| 不在第一阶段重构模型逻辑 | 第一阶段只统一入参、出参、身份、数据切片和证据链。 |

## 2. 目标状态

每个研究脚本最终应满足同一个运行契约：

```text
RunArgs / CLI
-> ResearchIdentity
-> DataSlice
-> ExperimentResult
-> artifacts
-> local experiment index
-> optional promotion package
```

统一后，一个脚本至少应能回答这些问题：

| 问题 | 统一来源 |
| --- | --- |
| 这个结果属于哪个研究主题？ | `ResearchIdentity.research_topic` |
| 这个结果的配置身份是什么？ | `ResearchIdentity.research_config_id` |
| 产物应该用什么文件名前缀？ | `ResearchIdentity.output_stem` |
| 输入数据切片是什么？ | `DataSlice` |
| 是否 PIT-safe？ | `DataSlice.pit_policy` / `lookahead_check` |
| 用了哪个 config 文件和关键配置？ | `ExperimentResult.config` |
| 输出了哪些 CSV / JSON / MD？ | `ExperimentResult.artifacts` |
| 核心指标是什么？ | `ExperimentResult.metrics` |
| 是否通过研究 gate？ | `ExperimentResult.gates` |
| 是否允许进入生产？ | `ExperimentResult.promotion.production_eligible` |
| 若 promotion，证据如何进入 registry？ | `PromotionCandidate` -> `promoted_registry.json` |

## 4. 统一数据结构

### 4.1 ResearchIdentity

现有 `scripts/research_identity.py` 已经提供：

- `slugify_token`
- `build_light_research_identity`
- `build_full_backtest_research_identity`
- `canonical_research_config`

建议保留这些函数，但让它们逐步生成同一类对象，或至少生成完全一致的字典。

建议字段：

```python
@dataclass(frozen=True)
class ResearchIdentity:
    result_type: str
    research_topic: str
    research_config_id: str
    output_stem: str
    canonical_config_name: str | None = None
    parent_result_id: str | None = None
```

兼容策略：

```python
identity = build_light_research_identity(...)
# 第一阶段仍返回 dict，新增 from_dict/to_dict 即可。
# 第二阶段再切到 ResearchIdentity 对象。
```

迁移原则：

- 不破坏现有测试和脚本调用。
- `research_config_id` 只由核心语义字段组成，不混入运行日期。
- `output_stem` 可以包含日期或 as-of 标签，但必须由 identity 统一生成。
- 所有产物都必须写入 `research_topic`、`research_config_id`、`output_stem` 三字段。

### 4.2 DataSlice

`src/models/data_slice.py` 已有切片 spec 归一化、哈希和 DataFrame 内容哈希函数。建议在此基础上新增 `DataSlice` 数据类。

建议字段：

```python
@dataclass(frozen=True)
class DataSlice:
    dataset_name: str
    source_tables: tuple[str, ...]
    date_start: str
    date_end: str
    asof_trade_date: str | None
    signal_date_col: str
    symbol_col: str
    candidate_pool_version: str | None
    rebalance_rule: str
    execution_mode: str
    label_return_mode: str | None
    feature_set_id: str | None
    feature_columns: tuple[str, ...] = ()
    label_columns: tuple[str, ...] = ()
    pit_policy: str = "unknown"
    config_path: str | None = None
    extra: dict[str, Any] = field(default_factory=dict)
```

统一哈希：

```python
def slice_hash(self) -> str:
    return hash_slice_spec(self.to_spec())
```

实际映射：

| 当前脚本字段 | 收口到 DataSlice |
| --- | --- |
| `start_date` / `end_date` | `date_start` / `date_end` |
| `daily_table` / `table_daily` | `source_tables` |
| `candidate_pool_version` | `candidate_pool_version` |
| `rebalance_rule=M` | `rebalance_rule` |
| `execution_mode=tplus1_open` | `execution_mode` |
| `open_to_open` 标签 | `label_return_mode` |
| `FEATURE_COLS` / `LABEL_COLS` | `feature_columns` / `label_columns` |
| `asof_trade_date` | `asof_trade_date` |
| PIT / coverage 说明 | `pit_policy` / `extra` |

### 4.3 ArtifactRef

目前各脚本都在手写：`summary_json`、`leaderboard`、`monthly_long`、`rank_ic`、`gate`、`doc`、`manifest` 等路径。建议统一成结构化 artifact。

```python
@dataclass(frozen=True)
class ArtifactRef:
    name: str
    path: str
    kind: Literal["csv", "json", "jsonl", "md", "parquet", "model", "directory", "other"]
    required_for_promotion: bool = False
    description: str = ""
```

常用 artifact 名称固定化：

| 标准名称 | 常见路径后缀 |
| --- | --- |

### 4.4 ExperimentResult

`ExperimentResult` 是所有研究脚本的统一出口。它不替代 CSV / MD 报告，而是登记“这次运行到底产出了什么”。

建议字段：

```python
@dataclass(frozen=True)
class ExperimentResult:
    result_id: str
    identity: ResearchIdentity
    script_name: str
    command: str
    created_at: str
    duration_sec: float | None
    seed: int | None
    data_slices: tuple[DataSlice, ...]
    config: dict[str, Any]
    params: dict[str, Any]
    metrics: dict[str, Any]
    gates: dict[str, Any]
    artifacts: tuple[ArtifactRef, ...]
    promotion: dict[str, Any]
    notes: str = ""
```

`result_id` 建议由稳定字段生成：

```text
{research_topic}:{research_config_id}:{primary_data_slice_hash}:{metrics_hash_prefix}
```

如果脚本是一次性报告、含运行日期，也可以在 `extra` 中记录 `as_of`，但不要让随机日期污染 `research_config_id`。

输出位置：

```text
data/experiments/research_results.jsonl        # 总索引
数据结果目录/{output_stem}_manifest.json       # 单次运行 manifest
```

写入策略：

- `manifest_json` 保持单次运行自包含。
- `research_results.jsonl` 作为轻量本地索引，便于后续查找 candidate 与 promotion 证据。
- `src/models/experiment.py` 的 `append_experiment_jsonl` 继续复用，只新增一层 `append_experiment_result()`。

## 5. 配置治理

### 5.1 配置分层保持不变

| 层级 | 路径 | 角色 |
| --- | --- | --- |
| 生产模板 | `config.yaml.example` | 只指向 promoted 默认方法。 |
| 本地配置 | `config.yaml` | 私有运行配置，可由 `QUANT_CONFIG` 覆盖。 |
| canonical 研究入口 | `config.yaml.backtest` | 当前研究主线回测入口。 |
| 历史研究快照 | `configs/backtests/` | 保存场景变体，不直接进生产。 |
| promotion registry | `configs/promoted/promoted_registry.json` | 唯一生产准入注册表。 |

### 5.2 增加 ConfigSnapshot

`ExperimentResult.config` 不应保存整个巨大 YAML 的无序复制，而应保存：

```python
{
  "config_path": "config.yaml.backtest",
  "config_hash": "...",
  "resolved_sections": {
    "paths": {...},
    "database": {...},
    "signals": {...},
    "portfolio": {...},
    "backtest": {...},
    "transaction_costs": {...},
    "prefilter": {...},
    "monthly_selection": {...}
  }
}
```

最低要求：

- 记录 `config_path`。
- 记录 `config_hash`。
- 记录脚本真正使用到的关键段，而不是只记录 CLI args。
- 如果 CLI 覆盖了 config，必须在 `params.overrides` 中明确列出。

## 6. promoted registry 的边界

`configs/promoted/promoted_registry.json` 不承担研究索引职责，只承担生产准入职责。

保留当前规则：

```text
未登记到 promoted_configs 的研究配置
不得写入 config.yaml.example 或 config.yaml 的生产默认主线。
```

建议新增的 promotion 流程：

```text
ExperimentResult(gates 全部通过)
-> 生成 PromotionCandidate 草案
-> 人工确认 owner_decision
-> 写入 configs/promoted/{config_id}.json 快照
-> 更新 promoted_registry.json
-> 更新 config.yaml.example 默认方法
-> 跑 promoted registry 相关 pytest
```

`promoted_configs` 可新增可选字段，不影响当前测试：

```json
{
  "source_result_id": "monthly_selection:...",
  "source_manifest": "data/results/..._manifest.json",
  "promotion_package": "configs/promoted/...json"
}
```

但必需字段仍保持当前 registry 已定义的：

- `config_id`
- `config_path`
- `promotion_date`
- `full_backtest_report`
- `oos_report`
- `state_slice_report`
- `boundary_report`
- `owner_decision`

## 9. Gate 字段标准化

不同脚本可以有不同 gate，但字段名应统一。

### 9.1 通用 gate

| 字段 | 含义 |
| --- | --- |
| `data_gate` | PIT、coverage、alignment 是否通过。 |
| `execution_gate` | 最新信号日、T+1 open、买入失败处理是否有效。 |
| `baseline_gate` | 是否超过指定 baseline。 |
| `rank_gate` | Rank IC / ICIR 是否达标。 |
| `spread_gate` | Top-K vs next-K / top-bottom spread 是否达标。 |
| `year_gate` | 年度切片是否无不可解释崩坏。 |
| `regime_gate` | strong-up / strong-down 等状态切片是否达标。 |
| `concentration_gate` | 行业集中度、单票集中度是否达标。 |
| `cost_gate` | 交易成本压力下是否仍有效。 |
| `governance_gate` | manifest、证据路径、owner decision 是否齐全。 |

### 9.2 promotion 约束

只有当所有 promotion 必需 gate 通过时，才能生成 `PromotionCandidate`。

```python
promotion = {
    "production_eligible": False,
    "registry_status": "not_registered",
    "blocking_reasons": ["cost_gate_missing", "owner_decision_missing"]
}
```

对于历史 rejected / frozen 方向，显式写：

```python
promotion = {
    "production_eligible": False,
    "registry_status": "rejected_or_frozen",
    "blocking_reasons": ["registry_candidate_status_rejected"]
}
```

## 12. 推荐的最小代码骨架

```python
# src/models/research_contract.py
from __future__ import annotations

from dataclasses import asdict, dataclass, field
from pathlib import Path
from typing import Any, Literal

from src.models.data_slice import hash_slice_spec


@dataclass(frozen=True)
class ResearchIdentity:
    result_type: str
    research_topic: str
    research_config_id: str
    output_stem: str
    canonical_config_name: str | None = None
    parent_result_id: str | None = None

    def to_json_dict(self) -> dict[str, Any]:
        return asdict(self)


@dataclass(frozen=True)
class DataSlice:
    dataset_name: str
    source_tables: tuple[str, ...]
    date_start: str
    date_end: str
    asof_trade_date: str | None
    signal_date_col: str
    symbol_col: str
    candidate_pool_version: str | None
    rebalance_rule: str
    execution_mode: str
    label_return_mode: str | None
    feature_set_id: str | None
    feature_columns: tuple[str, ...] = ()
    label_columns: tuple[str, ...] = ()
    pit_policy: str = "unknown"
    config_path: str | None = None
    extra: dict[str, Any] = field(default_factory=dict)

    def to_spec(self) -> dict[str, Any]:
        return asdict(self)

    def slice_hash(self) -> str:
        return hash_slice_spec(self.to_spec())

    def to_json_dict(self) -> dict[str, Any]:
        out = asdict(self)
        out["slice_hash"] = self.slice_hash()
        return out


@dataclass(frozen=True)
class ArtifactRef:
    name: str
    path: str
    kind: Literal["csv", "json", "jsonl", "md", "parquet", "model", "directory", "other"]
    required_for_promotion: bool = False
    description: str = ""

    def to_json_dict(self) -> dict[str, Any]:
        return asdict(self)


@dataclass(frozen=True)
class ExperimentResult:
    result_id: str
    identity: ResearchIdentity
    script_name: str
    command: str
    created_at: str
    duration_sec: float | None
    seed: int | None
    data_slices: tuple[DataSlice, ...]
    config: dict[str, Any]
    params: dict[str, Any]
    metrics: dict[str, Any]
    gates: dict[str, Any]
    artifacts: tuple[ArtifactRef, ...]
    promotion: dict[str, Any]
    notes: str = ""

    def to_json_dict(self) -> dict[str, Any]:
        return {
            "schema_version": "research_result_v1",
            "result_id": self.result_id,
            "identity": self.identity.to_json_dict(),
            "script_name": self.script_name,
            "command": self.command,
            "created_at": self.created_at,
            "duration_sec": self.duration_sec,
            "seed": self.seed,
            "data_slices": [x.to_json_dict() for x in self.data_slices],
            "config": self.config,
            "params": self.params,
            "metrics": self.metrics,
            "gates": self.gates,
            "artifacts": [x.to_json_dict() for x in self.artifacts],
            "promotion": self.promotion,
            "notes": self.notes,
        }
```

## 14. 这轮计划的最终收益

完成后，项目会形成一个轻量、可维护的研究治理闭环：

```text
脚本仍然独立、好跑、好改；
```
